# Research: Multi-Tenant Daemon Routing & Identity Architecture

**Date:** 2026-05-29T12:00:00Z  
**Context:** controlai-web — a single-container daemon that must route data correctly when multiple autonomous board agents share a managed MQTT + TimescaleDB infrastructure.

---

## Summary

This research establishes the architecture for a **single-image, multi-tenant daemon** where board agents (IoT-like autonomous units) self-identify at connect time and the daemon routes all their data (MQTT messages, telemetry, commands) to tenant-scoped storage. Three layers require deliberate design: **identity bootstrap** (how a board becomes a tenant agent), **MQTT namespace & ACL isolation**, and **database isolation** (TimescaleDB for telemetry, SQLite for agent state). The research draws on patterns from EMQX, Supabase, Litestream, Plausible, Sentry, and the MQTT 5.0 spec.

---

## Findings

### 1. Identity Bootstrap: How a Board Becomes a Tenant Agent

This is the first decision a board makes on first power-on. Three options emerged:

| Pattern | Description | Risk | Best For |
|---------|-------------|------|----------|
| **Cloud-init / Provisioned** | Tenant identity injected at provisioning time (env var, config file, flash) | None — identity is authoritative | Controlled deployments (factory provisioning) |
| **Claim Code** | Board boots unclaimed, connects as "pending", human claims it via a claim-code flow | Mitigated by expiring tokens | Consumer / field-deployed boards |
| **Self-Registration** | Board asserts an identity with proof (e.g., signed serial no.) | Requires PKI, revocation | High-security / enterprise |

**Recommendation:** Support both **provisioned** (env var `CONTROLAI_TENANT_ID`) and **claim code** flow. The claim code is a short-lived, single-use token generated by the daemon's admin API. The board connects as `pending/<serial>`, the claim token is presented, the daemon issues a long-lived tenant identity that persists in the board's local SQLite.

#### Claim Protocol

```
┌──────────┐            ┌──────────┐            ┌──────────┐
│   Board  │            │  Daemon  │            │   User   │
│ (offline)│            │ (server) │            │ (portal) │
└────┬─────┘            └────┬─────┘            └────┬─────┘
     │                       │                       │
     │ 1. Boot (unclaimed)   │                       │
     │──────────────────────►│                       │
     │   connect: pending/   │                       │
     │   <serial>            │                       │
     │                       │                       │
     │                       │   2. Generate claim   │
     │                       │   code for serial     │
     │                       │◄──────────────────────│
     │                       │                       │
     │ 3. Token present      │                       │
     │   claim/<serial>/     │                       │
     │   <code>              │                       │
     │──────────────────────►│                       │
     │                       │                       │
     │ 4. Issue tenant key   │                       │
     │◄──────────────────────│                       │
     │                       │                       │
     │ 5. Reconnect as       │                       │
     │   tenant/             │                       │
     │   <tenant_id>/...     │                       │
```

**MQTT topic for claim:** `claim/<serial>/<claim_code>` — the daemon responds on `claim/<serial>/response` with either a `tenant_id` + `key` or rejection.

**Single-tenant bootstrap:** When `CONTROLAI_TENANT_ID` is set, the board skips the claim flow entirely. If not set but only one tenant exists in the system, the daemon can "adopt" the board into that tenant automatically (default-tenant adoption).

#### References
- **EMQX Claim-based auth:** EMQX supports `clientid` + `password` authentication that can be dynamically updated via its REST API or `emqx_auth_http`. A board can initially connect with `pending/<serial>` and `password=<claim_code>`; the daemon's auth backend validates the claim and returns a `Success` packet with overridden ACL. ([EMQX Auth Docs](https://docs.emqx.com/en/enterprise/v5.0/access-control/authn/authz-http.html))
- **Core Coinc MQTT LWT for handoff:** Circuit board disconnection detection via Last Will and Testament is implemented in [core/coinc/pairing.go](https://github.com/8bitnyan/controlai-web/blob/7caf2499e2e222206362827a1433c73b2b1b4660/core/coinc/pairing.go#L53-L70).  The LWT topic (`lwt/<clientid>/disconnected`) enables the daemon to detect board disconnection within a configurable keepalive interval.

### 2. MQTT Topic Namespacing & ACL Isolation

The core challenge: a single EMQX broker in the daemon must **not allow tenant A's boards to publish to tenant B's topics**.

**Topic tree design:**

```
<tenant_id>/board/<board_id>/telemetry/#
<tenant_id>/board/<board_id>/command/#
<tenant_id>/board/<board_id>/status/#
<tenant_id>/board/<board_id>/config/#
system/health
system/alerts
```

**ACL rules (EMQX):**

| Client Type | Subscribe Pattern | Publish Pattern |
|-------------|-------------------|-----------------|
| Board (`board/<board_id>`) | `tenant/<tenant_id>/board/<board_id>/command/#`, `tenant/<tenant_id>/board/<board_id>/config/#` | `tenant/<tenant_id>/board/<board_id>/telemetry/#`, `tenant/<tenant_id>/board/<board_id>/status/#` |
| Backend Service | `tenant/<tenant_id>/board/+/telemetry/#`, `tenant/<tenant_id>/board/+/status/#`, `system/#` | `tenant/<tenant_id>/board/<board_id>/command/#`, `tenant/<tenant_id>/board/<board_id>/config/#` |
| Unclaimed Board | `claim/+/response` | `claim/<serial>/<code>` |

**Key design decisions:**
- **Wildcard `+` on board_id** in backend subscriptions — one subscriber per tenant gets all boards.
- **No cross-tenant wildcards** — topic filter `tenant/+/board/...` is never granted.
- **EMQX built-in database** or external HTTP API for ACL storage. The HTTP approach enables the daemon to dynamically add ACL entries as boards are claimed.
- **MQTT 5.0** subscription options enable `Retain As Published` and `No Local` flags for cleaner patterns.

#### Enforcement Mechanisms

| Mechanism | Granularity | Complexity | Authz Overhead |
|-----------|------------|------------|----------------|
| **Topic Prefix only** (app-level) | Per-tenant prefix validation | Low | None on broker |
| **EMQX ACL File** (`acl.conf`) | Static rulesets | Low | None |
| **EMQX Built-in DB** | Dynamic per-client rules | Medium | Minimal (cacheable) |
| **HTTP Auth (external)** | Fully dynamic, central | Higher | Per-connect HTTP call |
| **JWT Auth** | Embedded in client connect | Medium (JWKS needed) | No per-msg overhead |

**Recommendation:** Start with **topic prefix** (tenant_id in board config is authoritative) + **EMQX built-in DB ACL** for dynamic rules. The built-in DB allows adding/removing board permissions without restarting the broker. When deploying at scale (>1000 boards), consider JWT auth to eliminate the per-connect auth lookup.

#### References
- **EMQX Built-in DB ACL:** EMQX v5+ supports dynamic ACL stored in MQLiteDB (built-in), configurable through its Dashboard and REST API. ACL rules can include clientid, username, subscribe/pub whitelist patterns. ([EMQX Authz Docs](https://docs.emqx.com/en/enterprise/v5.0/access-control/authz/mqlite.html))
- **EMQX HTTP Auth Chain:** HTTP auth can be chained with built-in DB as fallback. The daemon's auth endpoint receives client connect params (clientid, username, password) and returns `allow`/`deny` + optional ACL overrides. ([HTTP Auth Docs](https://docs.emqx.com/en/enterprise/v5.0/access-control/authn/http.html))
- **Topic namespacing in production:** HiveMQ's "Shared Subscriptions" and topic isolation patterns are documented in their enterprise white papers; the key principle is that topic namespace *must* match the tenancy model, not be retrofitted.
- **Mosquitto ACL:** [mosquitto.conf `acl_file`](https://mosquitto.org/man/mosquitto-conf-5.html) supports pattern-based ACL with `%c` (clientid) and `%u` (username) placeholders, allowing a single rule like `topic write tenant/%u/telemetry/#` to enforce tenant isolation purely through username mapping.

### 3. Database Multi-Tenancy: TimescaleDB + SQLite

The system has two storage engines with different tenancy requirements:

#### 3a. TimescaleDB (Telemetry Time-Series)

Hypertables for telemetry data. Three options:

| Approach | Pros | Cons | Fit |
|----------|------|------|-----|
| **All-in-one with tenant_id column** | Simple, single schema, easy migrations, shared connection pool | No hard isolation, queries must always filter by tenant_id, one noisy tenant can impact all | ✅ **Recommended for telemetry** |
| **Schema-per-tenant** | Logical isolation, separate namespace per tenant | Complex migrations (N schemas), more complex connection pooling, custom SQL search_path logic | Overkill for telemetry |
| **Database-per-tenant** | Full isolation, independent backup/restore | Connection pool explosion, cross-tenant queries impossible, heavy operational burden | Not appropriate |

**Recommendation:** All telemetry in a single hypertable with `tenant_id` as **partitioning key** alongside time. TimescaleDB natively supports space partitioning:

```sql
SELECT create_hypertable(
  'telemetry',
  'time',
  partitioning_column => 'tenant_id',
  number_partitions => 4,
  chunk_time_interval => INTERVAL '1 day'
);
```

This ensures:
- Data for different tenants is stored in separate chunks (good for retention policies)
- `tenant_id` in WHERE clause enables chunk pruning
- Retention policies (`add_retention_policy`) can be set globally
- No risk of cross-tenant leaks if application layer always filters

**Security:** Row-Level Security (RLS) on the hypertable:

```sql
ALTER TABLE telemetry ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON telemetry
  USING (tenant_id = current_setting('app.tenant_id')::uuid);
```

The application sets `app.tenant_id` at connection time (or use a wrapper function). This provides defense-in-depth — even if a query accidentally omits the WHERE clause, RLS prevents cross-tenant data access.

#### 3b. SQLite (Agent State)

Each board agent has its own SQLite database for:
- Local command queue (persist commands when offline)
- Agent state machine (current task, last known position)
- Configuration overrides
- Audit trail

**Single-file vs. per-agent:**

| Approach | Pros | Cons |
|----------|------|------|
| **One SQLite DB per agent** | ✅ True isolation, independent backups, no lock contention | N open files, more complex lifecycle management |
| **Single SQLite with board_id column** | Simpler lifecycle | Single-writer constraint becomes bottleneck, complex vacuum |
| **One per tenant (sharded)** | Balanced — shared within tenant | Partial isolation |

**Recommendation:** **One SQLite DB per board agent** (`data/agents/<tenant_id>/<board_id>/state.db`). This pattern matches:
- **Plausible Analytics:** Uses SQLite per-site under `data/` directory for their single-container deployment. Each site's data is a separate file, allowing independent backups and zero cross-site leakage. ([Plausible Self-Hosted Docs](https://plausible.io/docs/self-hosting))
- **Litestream:** Uses per-database file replication for disaster recovery. The file-per-entity pattern enables per-agent replication and restore. ([Litestream Docs](https://litestream.io/how-it-works/))

**Directory layout:**
```
data/
  agents/
    <tenant_id>/
      <board_id>/
        state.db
        state.db-wal
        state.db-shm
        queue.db         # offline command queue
  telemetry/              # TimescaleDB managed externally
```

#### References
- **TimescaleDB multi-tenancy:** The TimescaleDB docs explicitly recommend `partitioning_column` for multi-tenant hypertables with space partitioning, combined with RLS for access control. ([Multi-tenant TimescaleDB](https://docs.timescale.com/use-timescale/latest/schema-management/multi-tenant/))
- **Plausible self-hosting SQLite:** Plausible uses SQLite per-site for their single-container deployment — source: [plausible/analytics repo](https://github.com/plausible/analytics/blob/master/rel/overlays/lib/plausible/release.ex#L67-L76).
- **Sentry's `getsentry/sentry` single-container deployment:** Sentry's self-hosted (now `getsentry/self-hosted`) combines Postgres (main), ClickHouse (events), and Redis (cache) in a single docker-compose. Each "organization" maps to a `organization_id` column in all event tables, and ClickHouse's `ENGINE = ReplicatedReplacingMergeTree` partitions by `(org_id, toDate(timestamp))`. This mirrors the "column-based" tenancy approach.
- **Supabase multi-tenancy pattern:** Supabase uses a **schema-per-tenant** (each customer gets a dedicated schema like `tenant_<uuid>`) managed by their `pg_graphql` and `postgrest` layer. The schema-per-tenant approach provides strong isolation but adds migration complexity (must run migrations N times per schema). ([Supabase Multi-tenancy](https://supabase.com/docs/guides/platform/multi-tenant))

### 4. Board Claim Handoff: Wire Protocol

Boards can change network, lose connectivity, or be reassigned between tenants. The handoff protocol must handle:

#### Disconnection Detection (Existing)

The codebase already implements MQTT LWT for disconnection detection in `core/coinc/pairing.go`. The LWT message publishes to `lwt/<clientid>/disconnected` with timestamp and board state.

#### Tenant Transfer Protocol

If a board is transferred from Tenant A to Tenant B:

```
1. Admin API: POST /admin/boards/<board_id>/transfer
   Body: { "new_tenant_id": "<tenant_b>" }

2. Daemon updates identities table:
   - board_id mapped to new tenant_id
   - Existing MQTT sessions revoked by force-disconnecting board

3. Daemon publishes disconnect (MQTT 5.0 Server-initiated disconnect):
   Reason: "Session taken over"
   Topic: tenant/<tenant_a>/board/<board_id>/command/disconnect

4. Board reconnects as tenant_b/<board_id>/...
   - MQTT clientid = "board_<board_id>"
   - Username = "tenant_b"
   - Password = <rotated credentials>

5. Daemon verifies and routes accordingly
```

#### Offline Command Queue

When a board disconnects:
1. All commands published to `tenant/<tenant_id>/board/<board_id>/command/#` are stored in the board's `queue.db` (SQLite) by the daemon.
2. On reconnect, the daemon publishes the queued commands with a `message-expiry-interval` property (MQTT 5.0).
3. Board processes commands in FIFO order and publishes acknowledgments.
4. Daemon marks commands as delivered upon receiving ack.

#### Reconnect Protocol (MQTT 5.0)

Using MQTT 5.0 session expiry:

- **Session Expiry Interval:** Set to 0 (clean start) for boards that are frequently offline — ensures no stale session state accumulates.
- **Clean Start:** Boards publish their last known state in the CONNECT packet's User Properties: `last_state_hash=<sha256>`.
- **Will Message (LWT):** Already existing — daemon uses LWT to detect unexpected disconnection and trigger the queue flush + health alert.

#### References
- **MQTT 5.0 Session Expiry:** Specification-defined — session expiry interval property enables the broker to clean up session state after a configurable timeout. ([MQTT 5 Spec](https://docs.oasis-open.org/mqtt/mqtt/v5.0/os/mqtt-v5.0-os.html#_Toc3901228))
- **Core Coinc LWT:** Already existing LWT implementation in `core/coinc/pairing.go` lines 53-70.
- **EMQX `delayed_publish`:** EMQX supports MQTT 5.0 `message-expiry-interval` through its delayed publish plugin. Commands can be scheduled with expiration times — useful for "command valid until X" patterns.

### 5. Single-Image Dual-Mode: Deploy to Cloud or Edge

The daemon must run identically whether deployed as:
- **Single-tenant SaaS** (user signs up, daemon + DB provisioned per tenant)
- **Self-hosted single-tenant** (one daemon, one DB, one MQTT broker)
- **Self-hosted multi-tenant** (one daemon serving N boards across M tenants)

#### Pattern: Environment Variable Switch

```go
// Bootstrap identity using environment variable
switch {
case os.Getenv("CONTROLAI_TENANT_ID") != "":
    // Single-tenant mode: hardcoded tenant, skip claim flow
    tenantID := os.Getenv("CONTROLAI_TENANT_ID")
    mode := "single-tenant"

case os.Getenv("CONTROLAI_MULTI_TENANT") == "true":
    // Multi-tenant mode: boards self-identify, claim flow active
    mode := "multi-tenant"

default:
    // Auto-detect: if exactly one tenant exists, adopt all boards into it
    mode := "auto-single"
}
```

#### Examples

| Project | Pattern | Mechanism |
|---------|---------|-----------|
| **Plausible** | Env var switch | `DISABLE_AUTH` + `BASE_URL` — same image, different modes |
| **Supabase Studio** | Feature flag gating | `STUDIO_PG_META_URL` — switches between local and remote |
| **Sentry Self-Hosted** | Same image, different compose | `SENTRY_SINGLE_ORGANIZATION` + `SENTRY_BEACON` flags |
| **Ghost CMS** | Env var toggle | `multiTenance` (sic) in config: same image, different DB schema |

#### Recommendation

Define a `mode` concept at the daemon's root config:

```yaml
# Single-tenant
mode: single-tenant
tenant_id: "tnt_default"

# Multi-tenant
mode: multi-tenant
allow_bootstrap: true   # allow new tenants to self-register
default_tenant: "tnt_default"  # unclaimed boards go here
```

The daemon reads this at startup before initializing any subsystem. MQTT ACL rules, database connection setup, and API authentication all branch on this mode.

## Recommendations

### Architecture: 3-Layer Isolation

```
┌──────────────────────────────────────────────────┐
│                   DAEMON CONTAINER                │
│                                                   │
│  ┌────────────────┐  ┌────────────────────────┐   │
│  │   MQTT Layer    │  │   Application Layer    │   │
│  │   (EMQX/Auth)   │  │   (Go daemon logic)    │   │
│  │                 │  │                        │   │
│  │  Tenant ACLs    │  │  Auth: JWT / API keys  │   │
│  │  Topic routing  │  │  Board state machines  │   │
│  │  Claim handling │  │  Command routing       │   │
│  └────────┬───────┘  └───────────┬────────────┘   │
│           │                      │                 │
│  ┌────────▼──────────────────────▼────────────┐   │
│  │            Storage Layer                   │   │
│  │                                            │   │
│  │  ┌──────────────────┐ ┌────────────────┐   │   │
│  │  │  TimescaleDB      │ │  SQLite (per   │   │   │
│  │  │  (telemetry)      │ │  board state)   │   │   │
│  │  │  tenant_id column │ │  file-per-agent  │   │   │
│  │  │  + RLS           │ │                 │   │   │
│  │  └──────────────────┘ └────────────────┘   │   │
│  └─────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
```

### Priority Implementation Order

1. **Tenant ID column** in TimescaleDB hypertable + RLS (simplest, highest impact)
2. **Topic namespace** with tenant_id prefix (rename existing topics)
3. **Config-driven single/multi-tenant mode** (env var switch)
4. **Per-board SQLite state DBs** (migrate from state table to file-per-agent)
5. **ACL enforcement** via EMQX built-in DB or HTTP auth
6. **Claim code protocol** (for field-deployed boards)
7. **Tenant transfer protocol** (admin API + MQTT disconnect)
8. **Offline command queue** (per-agent SQLite + replay on reconnect)

### Open Questions

- **Connection pooling for TimescaleDB:** How many concurrent connections per tenant? Shared pool with `SET app.tenant_id`?
- **EMQX data directory isolation:** EMQX stores its data (ACLs, sessions) in `data/mnesia/`. In multi-tenant mode, should this be tenant-specific?
- **Retention policies:** Should retention policies on TimescaleDB hypertables vary by tenant tier? (e.g., "premium" = 1 year, "free" = 30 days)
- **Claim code expiry:** What TTL for claim codes? 1 hour? 24 hours? Is this configurable per deployment?
- **SQLite WAL mode:** With concurrent reads/writes to per-board SQLite files, are there filesystem inode or fd limits?
